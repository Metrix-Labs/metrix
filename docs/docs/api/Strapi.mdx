# Metrix

:::info

Current state: **Stable**

:::

The Metrix class is the main object used in Metrix projects.
An instance of Metrix class is available as a global in any Metrix project: `global.metrix`.

## Class: Metrix

### `new Metrix(opts)`

- `opts`: <Type>Object</Type> Options that can be used on Metrix startup
  - `autoReload`: <Type>Boolean</Type> **Default:** true
    - If false, deactivate auto reload
    - If you modify any file in your Metrix project, it reloads your nodejs app
    - If any content-type is changed, it will reload the nodejs app
  - `serveAdminPanel`: <Type>Boolean</Type> **Default:** true
    - Should the admin panel be loaded and serve as a web client
    - The admin panel build will not be delivered if false
  - `appDir`: <Type>String</Type> **Default:** `process.cwd()`
    - The directory relative or absolute path where Metrix will write every file (schemas, generated APIs, controllers or services)
  - `distDir`: <Type>String</Type> **Default:** appDir value
    - The directory relative or absolute path where Metrix will read configurations, schemas and any compiled code

Instances of the Metrix class can be created using the new keyword.

Metrix extends the [Container](./container) class.

```javascript
const strapiInstance = new Metrix();
```

### `metrix.dirs`

- <Type>Object</Type>

Stored paths of file system.

- `dirs.dist`: <Type>[StrapiPathObject](#strapipathobject)</Type>
  - Build folder
- `dirs.app`: <Type>[StrapiPathObject](#strapipathobject)</Type>
  - Sources folder
- `dirs.static`: <Type>Object</Type> Define path to directories involving web client display
  - `public`: <Type>String</Type> Path to the folder to serve publicly (like files, images, etc..)

#### StrapiPathObject

- <Type>Object</Type>

A set of paths to specific Metrix project parts.

- `root`: <Type>String</Type> Root path
- `src`: <Type>String</Type> Sources route path to project files
- `api`: <Type>String</Type> Path to the folder containing project developers' API files (content-types, controllers, services, routes, etc..)
- `components`: <Type>String</Type> Path to the folder containing project developers' components
- `policies`: <Type>String</Type> Path to the folder where the Metrix project developers' policies are stored
  - A set of functions that check the state of the data and prevent the access to the API accordingly
- `middlewares`: <Type>String</Type> Path to the folder where the Metrix project developers' middlewares are stored
  - A set of function that wrap around routes and requests
- `config`: <Type>String</Type> Path to the folder containing project developers' config files

### `metrix.isLoaded`

- <Type>Boolean</Type>

  - `true`: Everything (all `register` and `bootstrap` functions available in your metrix project) has been loaded
  - `false`: There is something loading

Note: `register` functions are called before the `bootstrap` functions.

### `metrix.eventHub`

- [<Type>EventHub</Type>](EventHub)

The `metrix.eventHub` object is used to manipulate events within a Metrix project. It is an instance of the built-in EventEmitter class from Node.js, which provides a simple way to emit and listen for events.

The `metrix.eventHub` object is created using the `createEventHub()` function in the [EventHub](EventHub) module of the Metrix core. This function returns a new instance of the EventHub class, which extends the EventEmitter class and adds some additional functionality specific to Metrix.

#### Examples:

```javascript
// Listen for a 'user.updated' event and log the data
metrix.eventHub.on('user.updated', (data) => {
  console.log(`User ${data.id} has been updated`);
});

// Emit a 'user.created' event with some data
metrix.eventHub.emit('user.created', { username: 'johndoe', email: 'johndoe@example.com' });
```

In this example, we are emitting a `user.created` event with some data attached to it, and then listening for a user.updated event and logging the data. These events can be used to trigger actions within the Metrix application or to communicate with external systems.

For more information on how to use the EventEmitter class and its methods, see the [Node.js documentation](ttps://nodejs.org/docs/latest-v18.x/api/events.html#class-eventemitter).

### `metrix.log`

- [<Type>Winston</Type>](https://github.com/winstonjs/winston#creating-your-own-logger)

A logger provided by Metrix that uses the Winston logging library. It is the result of calling the `winston.createLogger()` function with the configuration defined by the user of the Metrix application.

The logger provides various methods for logging messages at different levels of severity, including error, warn, info, verbose, debug, and silly. The logging level can be set via the configuration to control which messages are logged.

#### Examples

```javascript
// Log an error message
metrix.log.error('Failed to start server', { error: err });

// Log a warning message
metrix.log.warn('Server is running in development mode');

// Log an informational message
metrix.log.info(`Server started on port ${PORT}`);

// Log a verbose message
metrix.log.verbose('Application state', { user: currentUser });

// Log a debug message
metrix.log.debug('API request received', { method: req.method, path: req.path });

// Log a silly message
metrix.log.silly('Entered loop', { count: i });
```

In these examples, we are logging messages at different levels of severity, including error, warn, info, verbose, debug, and silly. We are also passing in metadata as an object in the second parameter of each logging method.

The messages logged by metrix.log will be output according to the logging configuration set by the user of the Metrix application. This configuration determines which messages are logged and where they are logged (e.g. console, file, etc.).

### `metrix.cron`

### `metrix.telemetry`

- [<Type>TelemetryService</Type>](Telemetry)

The `metrix.telemetry` property provides access to the telemetry service instance. This service collects anonymous usage data about your Metrix application to help the Metrix team improve the product.

By default, the telemetry service is enabled, but you can disable it by setting the telemetryDisabled property to true in your application's package.json file, or by setting the `METRIX_TELEMETRY_DISABLED` environment variable to true. You can also disable telemetry programmatically by setting the isDisabled property of the `metrix.telemetry` instance to true.

### `metrix.requestContext`

- <Type>Object</Type> Context Storage

  - `run(store, cb)`: <Type>Function</Type>
    - `store`: <Type>Any</Type> Value that should be retrieved
    - `cb`: <Type>Function</Type> Callback
  - `get()` <Type>Function</Type>

The request context stores the ctx object from KoaJS on each request. This allows users to have access to the context from anywhere through the Metrix instance.

## Metrix containers

The metrix containers are accessible via `metrix.get('name-of-the-container')`.

### `config`

- <Type>Object</Type>

  - `get(path, defaultValue)`: <Type>Function</Type>
    - `path`: <Type>String</Type>
    - `defaultValue`: <Type>Any</Type>
    - Returns: <Type>Any</Type> - The value located at `path` or, if undefined, `defaultValue`.
  - `set(path, value)`: <Type>Function</Type>
    - `path`: <Type>String</Type> - Where the value should be stored
    - `value`: <Type>Any</Type>
  - `has(path)`: <Type>Function</Type>
    - `path`: <Type>String</Type>
    - Returns: <Type>Boolean</Type> - Does the `path` match a value stored in the config container.
  - `launchedAt`: <Type>Number</Type> **Default:** `Date.now()`
    Date in milliseconds when the server has started
  - `serveAdminPanel`: <Type>Boolean</Type> **Default:** `true`
    See [Metrix constructor](#new-strapiopts) options
  - `autoReload`: <Type>Boolean</Type> **Default:** `false`
    See [Metrix constructor](#new-strapiopts) options
  - `environment`: <Type>String</Type> - process.env.NODE_ENV
  - `uuid`: <Type>String</Type> - string extracted from `package.json` located in `metrix.uuid`
  - `packageJsonStrapi`: <Type>Object</Type> - object extracted from `package.json` located in `metrix` (except uuid)
  - `info`: <Type>Object</Type>
    - everything stored in the `package.json`
    - `metrix`: <Type>String</Type> - Current version of Metrix

Every file stored under the `config` folder will be injected in this config container object.
